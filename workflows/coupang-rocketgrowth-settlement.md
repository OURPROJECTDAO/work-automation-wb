# 워크플로우: 쿠팡 로켓그로스 정산 (coupang-rocketgrowth-settlement)

> 이 워크플로우를 건드리기 전 이 파일을 읽는다. 전역 함정은 pitfalls.md.

## 요약
- 쿠팡 로켓그로스(RG)는 쿠팡 자체 창고에서 직접 배송 — 우리 쪽 발주(포장·배송) 자체가 없어서
  **cheonnyeon-upload 파이프라인(발주자료 기반)에 애초에 안 잡힘**. 매달 별도 보충 입력 필요.
- **매달 반복.** 5월은 사용자 수작업(쿠팡로켓데이터집계v1.xlsx), 6월은 챗으로 첫 자동화(2026-07-02).
  **다음 실행 = 2026-08-01(7월분).** 아래 "다음 실행 시 체크리스트" 그대로 따라가면 됨.
- 최종 산출물 = 천년경영 파이프라인의 "쿠팡전체" 시트와 **동일한 컬럼 구조**로 만든 별도 xlsx
  (A일자·B관리코드·C상품명·D주문수량·E평균단가·F정산금액·G선결제택배비·H실제기입단가) — 사용자가 이걸 보고 천년경영에 수기 입력.

## ★ 다음 실행 시 체크리스트 (8/1, 7월분부터 매달 반복)
1. 사용자에게 쿠팡 WING **판매통계 export**(해당 월 1일~말일, 시트명 '판매통계') 요청 → 업로드받기.
2. **TOTAL행 제외**하고 파싱(마지막 행, A열='TOTAL'). 실수하면 정확히 2배로 잡힘 — 항상 이 필터부터.
3. 관리코드 매핑 우선순위로 시도:
   - ① **직전 회차(전월)의 최종 매핑**(이 워크플로우 산출물 또는 v1 원본의 "쿠팡상품관리"+"수량매칭" 시트) — 옵션ID가 같으면 그대로 재사용.
     ⚠️ "쿠팡상품관리" 류 export는 관리코드 칸이 **빈 문자열인 행**이 있을 수 있음 — `in dict` 멤버십 체크만 하면 못 거름,
     **truthy 체크** 후 안 되면 "수량매칭" 시트의 코드로 폴백.
   - ② 신규 옵션ID(리스팅 재발급 포함)는 옵션명 텍스트로 product_master 재검색(브랜드+규격 키워드).
   - ③ 그래도 안 되는데 **매출이 있는 것만** 사용자에게 가져가서 확인(매출 0인 미매칭은 무시, 다음에 팔리면 그때 확인).
4. **수량매칭(실제 개수 배수)**: 옵션명의 "n개" 패턴에서 추출. 기본은 n=낱개수 그대로지만,
   **그 상품이 쿠팡에서 박스/세트 단위로 팔리는 경우 n은 "박스 개수"를 뜻함**(예: 요구르트스파클링400ml은 24개들이 박스가
   기본 판매단위라 "24개"=박스 안 개수, 옵션 판매횟수 자체가 박스 판매횟수 — 그대로 낱개수로 안 쓰고 n을 곱해야 함).
   product_master 규격(예: 5L*3)의 "*N"이 곧 그 코드의 "쿠팡 판매단위"를 뜻하는 건 아님 — **옵션명에 박스개수가 이미 명시돼
   있으면(예: "24개") 그 값을 곧이곧대로 수량매칭으로 쓰면 됨**, 별도로 product_master 박스내품이랑 다시 곱하지 말 것
   (곱하면 이중계산). 애매하면 아래 5번 검산으로 판별.
5. **검산(필수, 이번 세션에 정착한 방법)**:
   - 관리코드별 내재단가(순매출÷실제수량)를 product_master **낱개 매출단가**와 대조 — 크게 다르면(2~3배 이상) 수량매칭이
     틀렸을 가능성. 박스매출단가 쪽에 더 가까우면 수량매칭을 다시 봐야 함.
   - 같은 관리코드에 옵션이 여러 개면(예: 1개/2개/3개 팩) 내재단가끼리도 서로 비교 — 정상 범위(대량구매 할인 감안 2~3배 이내)를
     벗어나면 의심.
   - 매핑 후 **관리코드별 순매출 합계가 원본 통계파일 TOTAL행 제외 합계와 정확히 일치**하는지 확인(매출 커버리지 100% 검증).
6. 수수료율 = **11.66%**(고정, 일반 쿠팡 마켓플레이스 12%와 다름). 바뀌면 사용자가 알려줄 것.
7. 최종 산출 — **[쿠팡전체] 단일 시트**로:
   - A(일자) = 그 달 특정 날짜로 **일괄**(사용자가 지정, 6월은 2026-06-28 사용 — 매달 발주일 재확인).
   - B(관리코드) · C(상품명, product_master 기준 ERP 상품명) · D(주문수량=실제수량, 매출 0인 코드는 행 자체를 뺌)
   - E(평균단가) = F/D 수식 · F(정산금액) = 순매출 합계(수수료 차감 전, gross) · G(선결제택배비) = **0**(로켓그로스는 쿠팡이 배송)
   - H(실제기입단가) = `F×(1-0.1166)/D` 수식 (F에 이미 차감 넣지 말 것 — H가 하는 일)
   - 채널은 쿠팡 로켓그로스 단일이라 다른 마켓 시트 안 만듦(cheonnyeon-upload 정규 파이프라인 27개 시트와는 별개 파일).

## 재사용 가능한 매핑 소스 (달마다 안 바뀌는 부분)
- 전월(또는 그 이전) 매칭본의 **"쿠팡상품관리" 시트**(옵션ID→관리코드) + **"수량매칭" 시트**(옵션ID→수량매칭 배수·기준마진율)는
  옵션ID가 유지되는 한 다음 달에도 그대로 재사용 가능. 매달 재확인 대상은 **신규 옵션ID**(신상품·리스팅 재발급)뿐.

## 계산식 요약
- 실제수량 = 순 판매 상품 수(옵션 판매횟수) × 수량매칭
- 정산금액(F) = 순매출 그대로(gross, 수수료 차감 전)
- 실제기입단가(H) = F × (1 − 0.1166) ÷ 실제수량
- 택배비(G) = 0 (로켓그로스는 쿠팡이 배송 부담)

## 코드 / 처리 방식
- **챗 네이티브(앱 미탑재)**. 코드 파일로 박제된 재사용 스크립트는 없음(sikbom_price_lookup.py처럼 반복 패턴이 더 굳어지면 고려).
- product_master는 `work-automation-app/reference/product_master.csv`에서 매번 최신본 읽을 것(월마다 갱신되므로).

## 검증 이력 (2026-06 첫 실행, 2026-07-02)
- 106개 옵션 중 103개 매핑(84개 기존 소스 재사용+19개 신규 확인), 나머지 3개는 매출 0.
- 매핑 후 순매출 합계 64,000,650원 = 원본 TOTAL행 제외 합계와 정확 일치(매출 커버리지 100%).
- 관리코드별 내재단가 편차 전수검사 — 박스/낱개 혼재 의심 케이스 0건.
- 최종: 관리코드 27개(매출 있는 것만), 실제수량 8,026개, 정산금액(11.66% 차감) 후 합계 산출.
- 산출 파일: `쿠팡전체_로켓그로스_202606.xlsx` (사용자가 천년경영에 수기 입력 예정, 2026-06-28 일괄 발주일).

# 워크플로우: intelligence-layer (지능 레이어 — 이력 엔진 + 두뇌: 마진·재고·A/B)

> 이 워크플로우를 건드리기 전 이 파일을 읽는다. 전역 함정 pitfalls.md, 설계근거 decisions/0018, 공유자산 manifest.md.
> ⚠️ **미구현 — 설계확정.** 직접 실행(코드/적재)은 다음 세션. **이 문서가 인계 정본.** (2026-06-12 설계)

## 0. 한 줄
관찰(대시보드·마진모니터·업로드감시)과 실행(가격변경·등록·발주·송장)은 **이미 다 있고**, 그 사이 '두뇌'(진단·추천·측정)가 빈칸. 매일/3년치 데이터를 private repo에 **이력으로 쌓아(이력 엔진)** → ① 마진 침식·제시 ② 입고·품절 예측 ③ 채널 가격 A/B 를 돌린다. 데이터는 부족하지 않다 — 안 모으고 있었을 뿐.

## 1. 큰 그림 (2층 + 루프)
- **루프**: 관찰 → 진단 → 추천 → 실행 → 측정 → (학습) → 관찰. 관찰·실행은 구축됨, 진단·추천·측정 = 두뇌(빈칸).
- **Layer 1 이력 토대(메모리)**: 과거 소급 4종 + 현재 앵커 1종 → private repo(`work-automation-data`) 날짜 파티션. 대시보드 parquet 패턴 재사용.
- **Layer 2 두뇌(결정)**: ①마진 침식·제시 ②입고·품절 예측 ③채널 가격 A/B.
- **실행 = 기존 actuator 재사용**: 가격변경 7채널(channel-margin-monitor)·상품등록·발주(logistics)·송장(invoice-fill).

## 2. ★중요 사실 / 정정 (헷갈리면 안 됨 — 사용자 명시)
- **(A) 정산 진실원천 = 매출자료** (천년경영 매출 업로드 정본 = 대시보드 영업이익현황 소스). 판매금액=정산. EasyAdmin·erp(천년경영업로드 output) 정산은 **raw 상류**(매일 추가정제 대상) → 진실 아님. "더 정확할 필요 없음". 따라서 채널별 정산내역서를 따로 파싱할 필요 없음.
- **(B) 택배 실배분 = 송장번호 그룹** (EasyAdmin). 같은 송장번호 = 한 박스(합포 구성). 대시보드 온라인 마진 탭의 **추정송장**(수량÷(합포수량×박스내품))·보정계수 k 를 **실제값으로 대체** → 상품별 마진이 추정→정확. 송장번호는 EasyAdmin 주문 export([10])에도 있어 별도 송장출력 파일 불요(백필도 커버).
- **(C) 매입가 두 출처**: master 매입단가(수정로그가 변경 기록) vs 유형별매입현황 실입고 단가. 마진모니터/실현마진은 **master 기준**. 차이 자체가 신호(매입현황↑인데 master 미수정 → 장부마진 과대, 곧 수정 시 마진 급락 = 조기경보).
- **(D) 마진모니터 정산식**(스마트 판매가×0.94 등)은 *앞으로 가격 정할 때*(미판매 리스팅) **추정용**. *실현* 마진 = 매출자료(정산) + 수정로그(매입가) + 송장(택배 실배분).
- **(E) Community Cloud 스케줄러 없음** → 알림은 GitHub Actions cron(repo 커밋 자료 diff→webhook) 또는 인앱 스냅샷 diff(지난 스냅샷 대비 입고/품절). 추이 저장 후순위였던 이유.

## 3. 관찰된 데이터 소스 카탈로그 (실물 확인 2026-06-12)

### 3.4 대시보드 매출 (영업이익현황 — 정산 진실원천) [기보유·적재됨]

### 3.6 erp입력자료 (천년경영업로드 output — velocity 보조) [기존 워크플로우]
- 파일: `{YYMMDD}.xlsx`. cheonnyeon-upload 산출. 27시트(14전체+13낱개).
- 전체시트: 일자·관리코드·상품명·주문수량·평균단가·정산금액·선결제택배비·실제기입단가. 낱개+개당수량·★기입수량·★낱개단가.
- 실제기입단가 = 정산금액×채널계수÷수량(식봄0.93·쿠팡0.88·스마트(정산+배송)/수량…). **마진 컬럼 없음**. 정산=raw.
- 용도: 채널별 일일 주문(velocity 보조). 단 3.5(EasyAdmin)가 더 우수(채널 명시·주문해상도·송장). 마진 진실 아님.

### 3.7 발주자료 (발주 demand — 리드타임) [기존: logistics 아카이브]
- 파일: `__발주자료{mmdd}.xlsx`. 1시트 '복사붙여넣기', ~129행, 8컬럼: erp관리코드·어드민옵션·총수량·평균단가·정산금액·판매처그룹·선결제택배비·옵션추가항목1.
- 용도: 채널분할 전 집계 주문. **리드타임 = 이 주문일 ⨯ 3.3 매입현황 입고일 조인**(데이터 보유, 조인만).

### 3.8 물류팀 전달 (보조 품절) [기존: logistics 출력]
- 파일: `물류팀_{mmdd}.xlsx`. 2시트. '물류팀'(프린트): 구분·규격·erp관리코드·어드민옵션·총수량·재고. '품절목록': 관리코드·상품명·발주수량·현재고(음수).
- 품절목록 = 발주 대비 부족(demand 기반, "오늘 못 채운 것"). 절대 품절/입고는 3.1 박스재고에서. 보조 신호.

### 3.9 송장출력 (택배 실배분 — PII heavy) [3.5와 중복]
- 파일: `__송장{mmdd}.xlsx`. 1시트, ~313행, 10컬럼: 상태·관리번호·발주일·판매처·주문번호·수령자·주소·상품명·택배사·송장번호. **PII 多**.
- 송장번호 그룹 = 합포 박스 구성(2B). 단 동일 매핑이 3.5(EasyAdmin 주문)에 있어 **별도 적재 불요**(백필도 3.5가 커버). 당일 신선 필요 시만.

## 4. 저장 설계 (private repo work-automation-data)
- 대시보드 parquet 패턴 재사용(token/repo 인자 주입, 페이지가 st.secrets). 신규 파티션(제안):
  - `history/price_changes.parquet`(또는 연파티션) — 수정로그 누적(상품코드·수정항목·수정전·수정후·일시).
  - `snapshots/stock_YYYY-MM.parquet` — 상품관리 일자 스냅샷(스냅샷일자·상품코드·관리코드·박스재고·박스매입단가·매입단가·매익률·매출단가). **✅ 적립 구현(1b)**, dedup키=(일자·상품코드).
  - `orders/easyadmin_YYYY-MM.parquet` — EasyAdmin 주문(PII 제거·19컬럼·송장그룹 해시·**기준일=발주일 우선**). **✅ 적재 시작(2026 3~5월 25,821행)**, ~8.5K행/월. core `orders.py`. 멱등=기준일 구간교체. 다음=Jan-Feb·2025 백필.
  - `purchases/buyin_YYYY-MM.parquet` — 유형별매입현황(관리코드·일자·수량·단가·거래처코드).
  - `derived/lead_time.csv` — 거래처/상품 리드타임(발주⨯입고).
- 매입가/거래처/매출 = 영업기밀 → public app repo 금지, private data repo.

## 5. 구현 단계 (세부)
### Phase 1 — 이력 토대 (대부분 보유 데이터/조인)
- **[첫 브릭] 1a 수정로그 적재 — ✅ 완료(2026-06-15)**: 파서(openpyxl, read_only 금지) → 매입단가·매출단가 행 필터(2,567건=매입2009/매출558) → 정규화(상품코드 NFC 문자열키·수정전/후 콤마제거 float·수정일자 datetime, **처리자 PII 제외**) → `work-automation-data:history/price_changes.parquet` PUT. **dedup키=(상품코드·수정항목·수정일자), 중복0·멱등.** 3년 backfill 불가(1년 롤링) → **월 누적으로 전진 축적.** 역재생 `as_of_value`+`current_purchase_price` 결선 — **앵커=낱개 매입단가[8] 확정(95.4%, 880/922; 박스=낱개×박스내품 917/917).** 실증 OK(코카콜라제로 18변경 as-of: 변경전=최초수정전·현재=최신수정후 일치). 다음=1b 상품관리 스냅샷. core: `core/intelligence/price_history.py`.
- **1b 상품관리 스냅샷 — ✅ 완료(2026-06-15)**: 상품관리 업로드 훅(`app/pages/3_연동데이터관리/1_상품관리.py` `_accumulate_snapshot`)이 **새 업로드 df를 날짜본으로 적립**(비차단·st.toast·master 저장과 독립). 스냅샷일자=파일명 `Exp{YYMMDD}` 추출일(없으면 당일). 파티션=`snapshots/stock_YYYY-MM.parquet` 7컬럼(스냅샷일자·상품코드·관리코드·박스재고·박스매입단가·매입단가·매익률·매출단가 = 위치 [3][4][14][9][8][12][10]). dedup키=(스냅샷일자·상품코드) keep=last 멱등. 전이 `detect_transitions`(박스재고 0이하↔양수=품절/입고). core: `core/intelligence/stock_history.py`(price_history 동형). **골든: 실 product_master 4340→4340·키중복0·결측0·parquet 라운드트립·dedup 멱등·전이 정확.** ⚠️ core 신규 import → 첫 배포 후 **Reboot app** 1회. **forward 축적**(과거 소급 불가); 첫 업로드=스냅샷#1, 둘째부터 전이. 다음=두뇌① 마진 침식 or 1c 리드타임.
- **1c 리드타임**: 발주(3.7)⨯매입현황(3.3) 조인 → lead_time. 소진 예측 입력.
- **1d 채널 listing 스냅샷 보관**: 마진모니터 listing CSV 갱신 시 날짜본 적립 → 채널 가격 이력(A/B 기반). (현재도 덮어쓰는 중 — 상품관리와 동일)
- **1e 행사 로깅 시작(forward only)**: 가격변경/행사 이벤트 기록 테이블 신설(백필 불가, 지금부터만).
### Phase 2 — 진짜 마진 (대부분 이미 해결)
- 실현 마진 = 매출자료(3.4 정산, 보유) + 수정로그(3.2 매입가) + 송장 실배분(3.5). **별도 채널 정산내역서 파싱 불요(2A)**.
- **송장 실배분**: EasyAdmin 주문(3.5) 적재(PII 제거) → 송장번호 그룹 → 박스당 실택배비를 구성 상품에 배분 → 상품별 택배비 → 상품별·채널별 실현 마진. **대시보드 추정송장·k 대체(2B)**. core: `core/intelligence/ship_alloc.py`.
### Phase 3 — 시장 지능 (별도 트랙, net-new)
- 경쟁가 크롤링(네이버 최저가·쿠팡) + 관리코드 퍼지 매칭. 브리틀 → 핵심 상품부터 점진. 마진 추천을 시장대비로. (사용자: 없지만 끌림)
### Phase 4 — 퍼널 (가능 채널만)
- 리스팅 통계(조회·전환) 캡처(주는 채널만). 광고비·반품 붙이면 진짜 단위경제. 부분 데이터 → 전체 의존 금지. (사용자: 일부 채널만 가능)

## 5.6 웹앱 통합 (UI/페이지 — 어디에 어떻게 들어가나)
> Streamlit st.navigation + st.Page(명시 등록 필수, 자동탐지 안 됨). core 수정분은 Reboot app 1회(pitfalls). 토큰=st.secrets [data] pat/repo 주입.

**A. 데이터 적립 (입력 UI)**
- **이력 적립 페이지(신규)** — 연동데이터관리 그룹에 추가(상품관리 옆). 탭: ①수정로그(3년 백필 업로드 + 이후 주기 업로드) ②유형별매입현황 ③EasyAdmin 주문(확장주문검색 .xls). 각 탭=파일 업로드→파싱 미리보기→[적립] 버튼(private repo 파티션 PUT)→결과 건수. 대시보드 '데이터 추가' 탭 패턴 재사용(증분 업로더).
- **상품관리 스냅샷=자동** — 기존 상품관리 업로드 페이지(`app/pages/3_연동데이터관리/1_상품관리.py`)에 *덮어쓰기 직전 스냅샷 적립* 한 줄만 추가. 별도 UI 없음(사용자는 평소처럼 업로드).

**B. 알림 (출력 UI — 스케줄러 없음 대응, 2E)**
- **앱 상단 배너** — 앱 열 때 "지난 스냅샷 대비: 입고 N · 신규 품절 M · 마진 침식 K건" 요약 + 해당 페이지 링크. 인앱 스냅샷 diff(푸시 아님, 인프라 0).
- **푸시(선택)** — GitHub Actions cron이 매일 커밋 자료 diff → 카카오/Slack/메일 webhook. 앱 밖(별도 트랙).

**C. 두뇌 3종 (보기 UI)**
- **① 마진 침식·제시** → channel-margin-monitor 페이지에 **'마진 추이' 탭 추가**: 상품/채널 마진을 기간별 라인 + 침식 경보 표(판매가 정체 + 매입가↑ 상품). 기존 8채널 모니터 옆 탭.
- **② 입고·품절 예측** → upload-monitor 페이지(`app/pages/7_업로드감시.py`) 확장: 기존 매트릭스에 '소진 예측일·재발주 시점' 컬럼/탭 추가(리드타임 1c). 또는 신규 '재고 지능' 페이지.
- **③ 채널 가격 A/B** → **신규 페이지 '가격 A/B'**(분석/등록 그룹): 상품 선택 → 채널별 가격 vs 판매량 비교표 + 가격변경 전후 그래프. EasyAdmin velocity·송장 실배분 기반.
- **온라인 상품마진(실측 교체)** → 대시보드 온라인 상품마진 탭(현재 추정송장·k)을 *실측(송장 실배분, 2B)으로 교체*. 페이지 동일, 계산만 정확화(Reboot 불필요 가능—core면 필요).

**D. 네비게이션 요약**
- 신규 페이지: '이력 적립'(연동데이터관리 그룹) · '가격 A/B'(분석/등록 그룹) · (옵션)'재고 지능'.
- 확장: 마진 모니터(마진 추이 탭) · 업로드감시(예측 컬럼) · 대시보드(온라인마진 실측·상단 알림 배너).
- 전부 st.Page 명시 등록. 새 page만 자동반영, core import 모듈 변경은 Reboot app 1회.

## 6. 두뇌 3종 (capability)
- **① 마진 침식·제시 — ✅ v1 구현(2026-06-15)**: **채널 baseline 기준**(base 매익률=오프라인이라 폐기, 사용자 확정). 신호=(최근 3개월 매입단가 인상, 가격이력 1a) ∩ (채널 마진 < 기준마진, cmm 탐지<-0.01). **이미 재설정한 건 자동 제외**. 8채널 통합 작업목록 + 기준 회복 권장가(cmm 재사용). 신규 page `8_마진침식.py` + core `margin_erosion.py`. 조인키=관리코드. 검증=89관리코드 인상→침식 50건(게토레이 4채널 동시·동원참치 +30%). 한계=합성코드(합포/PC/소분) 미조인. 향후=②velocity 가중·실현마진(송장)으로 정밀화.
- **② 입고·품절 예측**: 박스재고 스냅샷 추이(1b)+실입고(3.3)+velocity(3.5) → 소진 예측 + 리드타임(1c) 재발주 시점. upload-monitor 연계(반응형→예방형).
- **③ 채널 가격 A/B**: EasyAdmin(3.5) 채널별 가격·velocity → 채널 간 자연실험 + 전후비교(가격변경 이벤트/listing 이력 기준). 송장 실배분으로 채널별 실현마진까지. ★준실험(계절·행사 교란 → 1e 행사 로깅으로 보정). RCT 아님(전표 고유ID 없음·수동 다운로드).

## 7. 전용 함정
- EasyAdmin .xls = HTML(read_html), 헤더 첫 행(컬럼 숫자로 잡히면 header=0/행탐색). PII 제거 필수(3.5 목록).
- **정산 raw 혼동 금지** — 매출자료(3.4)가 정산 진실. EasyAdmin/erp 정산은 raw.
- 매입가 두 출처(2C) 구분 — 마진모니터=master.
- 재고 절대값은 흐름누적 오차(이월·조정·누락) → 상품관리 스냅샷 앵커.
- 송장번호=식별자 → 그룹키/해시로만, 원본 미저장.
- 스케줄러 없음(2E) → 알림은 Actions cron 또는 인앱 diff.
- 매입현황 일자 stray값(파싱 이상치 1건 확인) 검증.
- 매입단가 소수 정상(예 1521.5833); 키 매칭 시 상품코드 NFC·정수 ID float→int(전역 pitfalls).

# 로그: 발주서출력업무 — 합포 감지를 어드민옵션 대괄호 기준으로 교체 (영문/PC코드·N개)

## 무엇
다른 상품 합포 분리(`split_multiproduct_cells`)의 감지 기준을 `옵션추가항목1` 숫자코드 패턴에서
**어드민옵션의 `[erp코드]` 대괄호**로 변경. 영문접두/PC 코드 합포까지 분리되도록 수정.

## 왜
사용자가 0608 raw(`판매처상품매출통계_...070749`) + 1차 정제 산출물을 제출.
정제본 61행이 안 나눠짐: `명가 꽈배기 참깨[MGG1EA-67-74]`+`흑당[MGG1EA-67-74-01]` 합포인데 1행으로 뭉침.
같은 파일의 코카콜라 합포(`[31-03-05][31-22-02]`)는 정상 분리됨.

## 원인
- `_ERP_CODE_RE = re.compile(r'\d{2}-\d{2}-\d{2}')` — **숫자전용**.
- `MGG1EA-67-74`/`MGG1EA-67-74-01` 같은 영문접두 코드는 미매칭(끝 `67-74-01` 1개만 잡혀 2개 조건 불만족).
- 즉 숫자코드 합포만 분리, 영문/PC(`MGG1EA-`, `PC005708`, `BT10EA-` 등) 합포는 전부 누락.
- 감지를 구분자 없이 붙은 `옵션추가항목1`(`MGG1EA-67-74MGG1EA-67-74-01`)에서 한 것도 취약.

## 변경 (work-automation-app, commit 4f62f82)
- `core/workflows/logistics_order.py`:
  - 상수 `_BRACKET_CODE_RE = re.compile(r'\[([A-Z0-9][A-Z0-9.\-]*)\]')` 추가. `_ERP_CODE_RE`는 fallback용 유지.
  - `split_multiproduct_cells` 재작성:
    - 1순위: 어드민옵션 `[erp코드]` 대괄호(숫자 포함 코드만) **2개 이상** → 코드 = 대괄호 캡처, 어드민옵션을 각 대괄호 끝 경계로 N분할, 총수량·정산금액 ÷ N, 각 행에 해당 erp코드 직접 기입.
    - 2순위(구버전 호환): 대괄호 2개 미만이면 `옵션추가항목1` 숫자코드 2개 fallback(기존 로직).
  - 기존 2개 한정 → N개 일반화. erp관리코드는 None+fill 대신 직접 기입(포맷 무관 안전).

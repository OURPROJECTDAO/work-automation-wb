# 2026-06-16 채널 내 고객키 + 합포박스키 설계 (탐색·발견 — 코드/적재는 다음 세션)

## 무엇 / 왜
두뇌③ 채널 가격 A/B·코호트·재구매 분석 + **합포 그룹핑 정밀화(ship_alloc 과거 데이터)** 를 위해, EasyAdmin 주문에 **채널 내 고객키 + 합포박스키**(둘 다 비-PII 해시)를 박제하려는 설계. 사용자 아이디어. 적재된 orders parquet은 PII 전제거됐으나 **원본 .xls엔 풀 PII 살아있음** → 원본 재처리 시 해시키 박제 가능(송장그룹 해시 패턴 확장).

## 동기 (직전 합포 논의에서 파생)
- 탭D(데일리)는 송장출력 풀 PII로 175~200 합포를 **수령자 그룹 정밀 재구성**(2026-06-16 완료). 그러나 과거(ship_alloc·P2)는 orders parquet에서 수령자·주소·주문번호 제거 → 송장그룹(해시)만 남아 **175~200 물리합포(송장 분리)를 못 잡음**(로그상 "98.6% 단일라인"에 위장 섞임). 채널 총택배비는 00-12 정합으로 맞으나 **합포 상품 ↔ 비합포 상품 상대 배분 왜곡** → 두뇌③ 채널 단위경제 흔듦.
- 해법 = 적재 시 (수령자+주소)로 **합포박스키** 박제 → 과거도 정밀 합포. 같은 재처리에서 (수령자+전화)로 **고객키** 도 뽑아 코호트 토대.

## 두 키 (용도 분리)
- **고객키** = 수령자명 + 전화 → 채널 내 재구매·코호트·A/B 고객군. (전화가 고객 식별 안정, 주소는 이사로 변동)
- **합포박스키** = 수령자명 + 주소 (+ 발주일) → 합포 그룹핑(ship_alloc·탭D 과거판). (같은 배송지=한 박스)
- **★ 채널 내 키** (채널명 prefix). 채널 간 동일인 매칭(엔티티 레졸루션)은 신뢰도 낮아 **후순위 별도 레이어** — A/B 채널군 비교엔 채널 내로 충분.

## 안전 원칙 (고정)
- `해시( 정규화(조각들) + 솔트 )`. **솔트=repo 시크릿**(st.secrets). 결정적(같은 고객=같은 키)이되 솔트 없이는 역추적 불가.
- **원본 PII는 키 만들고 즉시 폐기.** parquet엔 해시 컬럼만. 송장번호 해시(_hash_invoice) 패턴 그대로.

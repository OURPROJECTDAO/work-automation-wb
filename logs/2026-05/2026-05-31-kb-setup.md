# 2026-05-31 · KB 기억 시스템 구축 (Drive 탐색 → GitHub 이관)

## 무엇
업무 자동화 프로젝트의 영속 기억(KB) 저장소를 세우고, Drive 시도 후 GitHub로 확정·이관.

## 왜
채팅은 매 세션 리셋. 상태·결정·함정·로그를 채팅 밖에 구조적으로 쌓아 "읽기-먼저/쓰기-나중"으로 이어가려고.

## 변경 (경위)
- 저장소 후보 검토: GitHub(투자 검증됨) vs Drive(이미 연결+엑셀 colocate) vs 단일문서.
- Drive 선택 → 실측: 읽기/쓰기/원본 왕복 OK. 단 커넥터가 생성+읽기만(update/delete 없음) 발견 → append-only 타임스탬프 스냅샷 + read-newest 설계로 대응.
- 투자시스템 에이전트 피드백 반영(부트로더 v2): 풀스냅샷(델타 금지)·폴더나열후 최신선택·UTF-8 디코딩·검증 단계화·PII 금지·payload 승인.
- 실사용 캐치업 지연 발견(폴더마다 나열→탐색→다운로드, 호출 7~10번 순차) → GitHub 전환 결정(decisions/0001).
- GitHub 이관: private repo OURPROJECTDAO/work-automation-wb 생성, fine-grained PAT(contents R/W). bash+PAT contents API로 INDEX/state/pitfalls/roadmap/decisions/logs 작성. 고정 파일명이라 read-newest 곡예 제거.
- 엑셀 작업물은 Drive(templates/inputs/outputs)에 유지.

## 검증
- PAT 접근 HTTP 200, repo private/main 확인. 파일 PUT 후 트리 재조회 + 캐치업(INDEX+state+pitfalls 1회 읽기) 실측.

- Drive의 KB용 폴더(index/state/pitfalls/roadmap/decisions/logs)는 폐기 가능(사용자 수동 삭제). 엑셀 폴더는 유지.

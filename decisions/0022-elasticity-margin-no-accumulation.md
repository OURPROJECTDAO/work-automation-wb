# 0022 — 탄력성 = 마진율별 판매량 · forward 적립 거부

날짜: 2026-06-16 · 상태: 채택 · 관련: 0018(intelligence-layer)·0020(로드맵)·workflows/intelligence-layer.md §6 ③

## 맥락
두뇌③ 채널 가격 A/B의 잔여 = 탄력성. 정석 설계는 가격↔판매량 + 토대(1d 채널 listing 스냅샷·1e 행사
로깅)를 forward로 쌓아 교란 보정. 사용자와 범위를 좁히며 여러 갈림길을 정리.

## 결정 (사용자 확정)
1. **레버 = 마진율 (가격 아님).** "당시 마진율 얼마에 얼마 팔렸다"가 실무 감이 더 좋음 — 사용자가 실제로
   돌리는 게 마진율(채널마진모니터·360 목표마진). 마진율 = 매출자료 실현마진(순이익÷매입가, 진실원천).
2. **윈도우 = 최근 18개월 고정.** 그 전은 무의미.
3. **forward 적립(1d listing 스냅샷·1e 행사 로깅) 안 만든다.** 과거 판매가·마진이 이미
   easyadmin_*.parquet(판매가)·매출자료(실현마진)에 다 있음 → 별도 적립 불필요. 과거 데이터 노이즈를
   이유로 안 쓸 필요 없음.
4. **행사 감별 안 한다.** 행사 탄력성은 없다고 침 → 교란 보정 불요.
5. **품절 보정 안 한다 ("그냥 둔다").** 과거 재고 이력이 신뢰성 있게 없음(1b 스냅샷 forward-only·2026-06~,
   품절목록 미적립) → 과거 품절을 못 잡음. 억지 감별은 틀린 신호를 진짜처럼 만들어 해로움. 아래로 튄 점은
   사람이 판단.

## 근거
- 마진율은 사용자의 의사결정 단위이자 채널 수수료/매입가 변화를 흡수하는 정규화 축 → 채널 간·시점 간 비교 정직.
- 적재된 backbone(매출자료 42만·EA 주문 39개월·매입현황 53파티션)만으로 마진율↔판매량 월별 시계열이
  이미 나옴 → forward 적립 ROI 낮음(밀도도 갱신 빈도 의존).
- v1은 인과(RCT) 아님을 명시 — 상관/서술. 정밀화(교란 보정)는 데이터가 안 받쳐주므로 미루는 게 정직.

## 결과
- `10_가격AB.py` 하단 '마진율별 판매량'(page-only): (관리코드×채널) 월별 점=실현마진율→판매량(낱개)
  산점도 + 구간별(역마진/0–8/8–12/12–16/16%+) 평균 월판매량. core 무변경.
- 두뇌③ 완료(서술+탄력성). 두뇌 3종+통합 상품360 카드 전부 v1.
- **재검토 트리거**: 나중에 정밀 탄력성(가격변경 준실험·교란 보정)이 필요해지면 그때 1d/1e를 forward로
  켜는 것부터 선행(지금은 거부). 품절 정밀 표시는 1b 재고 스냅샷이 충분히 쌓인 *미래* 구간에만 가능.

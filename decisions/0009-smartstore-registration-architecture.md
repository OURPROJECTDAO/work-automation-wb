# ADR 0009 — 스마트스토어(멀티채널) 상품 등록: 챗 네이티브 + 재사용 엔진/캐시

날짜: 2026-06-09
상태: 채택

## 맥락
상품 등록 업무를 한 바퀴 돌려봄(스마트스토어, 글로벌하베스트 15개 배치). 약 7~8개 채널로 확대 예정. 운영 형태를 ① 풀 Streamlit 템플릿화 ② Streamlit+Claude API ③ 챗 네이티브+KB 중 결정 필요.

## 결정
**챗 네이티브 + KB**로 운영. 단,
1. 결정적 코어(코드/가격/배송비/이미지/서식)는 **채널 config로 구동되는 재사용 엔진**으로 추출(향후).
2. **관리코드 → {정제상품명, 카테고리코드} 캐시를 채널별로 축적** — 신규 상품만 AI/검색, 재등록은 결정적.
3. 캐시가 충분히 차면 결정적 재업로드를 Streamlit 버튼/Claude API로 분리(지금 미구현, 데이터 쌓인 뒤 재검토).

## 근거
- 워크플로우 = 결정적 90% + 판단 10%(상품명 정제·카테고리 매핑). 판단 부분은 AI/웹검색 필요 → 풀 Streamlit은 핵심을 자동화 못 함(사람 수기).
- 챗은 결정적(코드 실행) + 판단(AI 제안·검색·대화형 수정) 둘 다 처리. 기존 아키텍처(Streamlit=무LLM 생산 / Claude=판단)와 일관.
- 채널 확장 = 새 코드가 아니라 config + reference + 캐시 한 세트 추가 → 확장 비용 낮음.
- 캐시 전략으로 AI 비용은 "신규 상품 1회"로 수렴, 카탈로그 안정 시 자동화율 상승.

## 대안 (기각)
- 풀 Streamlit: 상품명·카테고리 수기 부담 + 7~8채널×양식별 코드 유지부담.
- Streamlit+Claude API: API키·비용·배치형 신뢰성↓(이름/카테고리는 대화형 수정 유리). 볼륨 커지면 재검토.

## 영향
- workflows/smartstore-register.md 신설. 운영은 챗, 규칙·reference·캐시는 KB.

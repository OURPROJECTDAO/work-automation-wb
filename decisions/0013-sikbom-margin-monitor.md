# ADR 0013 — 식봄 마진 모니터: 스마트스토어 표준 정산식 + 합포량 N 외부 reference

상태: 채택 (2026-06-11)

## 맥락
채널 마진 모니터를 식봄(2번째 채널)으로 확장. 식봄 골든(식봄결과페이지)의 정산식은 스마트스토어와 두 군데 다름:
1. 실택배비를 **합포 N으로 분기**(N<2 → 3000, N≥2 → 3700; 묶음판매 +700).
2. 권장가 **ROUND(반올림)**.
또한 스마트스토어는 이미 원래 엑셀의 3000/3700을 폐기하고 **2700 단일·올림(ceil)** 으로 단순화한 바 있음(2026-06-10).

식봄 다운로드 구조 차이: 배송비 숫자 없음(상수 필요)·즉시할인/포인트/바코드 컬럼 없음·합포량(N)은 다운로드에 없고 별도 파일(상품몇개합포인지 = 골든 마진율예외처리시트)에만 존재.

## 결정
1. **정산식 = 웹앱 스마트스토어 표준을 단일 기준으로.** 새 채널은 골든이 다른 값을 써도 **실택배비 2700 flat·권장가 올림(ceil)** 을 따른다. 채널별로 두는 것은 **수수료율(commission)** 과 **배송비 입력 방식**뿐. (식봄 수수료 7%.)
   - 결과: 식봄 마진율은 골든보다 (3000−2700)/정산액(합포≥2는 (3700−2700)/정산액)만큼 높게 나옴 — 의도된 표준화, 버그 아님. 골든은 입력(정산가·매입가·N) 검증용으로만 사용.
2. **배송비 income측 = 상수 3000**(식봄). 다운로드에 숫자가 없어 config `ship_fee_const`로 주입. 정산가 = net×0.93 + 3000×0.967. (폐기 대상인 cost측 3000/3700과 별개 — 이건 정산수입측.)
3. **합포량 N = 신규 공유 reference `hapo_multiplier.csv`**(상품번호·합포량·채널). 바코드 없는 채널 공용(스마트스토어 판매자바코드의 외부판). 상품번호 단일키(플랫폼마다 고유)로 채널 무관 조회, 미등록 1, 분수 허용. config `n_source`("download"/"ref")로 채널별 선택.
4. **코어 일반화는 config 축 2개로 최소화**: `ship_fee_const`, `n_source`. parse_download는 누락 컬럼을 0/None 허용. compute 구조(2700·ceil·net×(1−수수료)+배송비×0.967)는 불변 → 스마트스토어 회귀 보존.

## 근거
- 단일 표준 정산식은 채널 증가 시 유지보수 단순(채널별 수수료만). 골든 충실은 채널마다 분기 폭증.
- 합포 +700·ROUND는 이미 스마트스토어에서 정책적으로 접은 항목 → 식봄만 되살릴 이유 없음(사용자 확정).
- N 외부 reference는 "모든 플랫폼 공용(바코드로 끌어오는 채널 제외)"이라는 업무 실태와 일치. 한 파일이 전 채널 N 공급.

## 결과 / 영향
- compute() 구조 무변경 → 스마트스토어 705/706 그대로.
- 식봄 골든 입력 대조: 정산가 485/485·N 485/485·매입가 480/485(vintage). 로직 불일치 0.
- 향후 채널: CHANNEL_CONFIG 한 세트(+수수료·배송비 방식·N 출처). 정산식은 고정.
- 미해결: 식봄 가격 일괄변경 양식(다운로드 양식 컬럼맵 별도). baseline↔product_master 조인 갭.

## 대안 (기각)
- 골든 충실(채널별 실택배비 분기·권장가 라운딩): 유지보수 복잡·정책 일관성 훼손.
- N을 채널별 (채널,상품번호) 복합키: 상품번호가 플랫폼 고유라 불필요. 단일키로 충분.
